# Fragment 交互架构

## 1. 产品主线

```text
Box 投递
  ↓
等待变化与低频回响
  ↓
个人碎片轨道
  ↓
个人脑洞网络
  ↓
社区树状延申
  ↓
收进 Box / 继续写
```

## 2. 分层设计

| 层级 | 当前职责 | 后续替换方向 |
| --- | --- | --- |
| UI | `Index.ets` 中的 Box、捕捉面板、个人轨道/关系图、社区延申树 | 拆分可复用组件与多设备响应式布局 |
| Model | 页面内的 FragmentItem、GraphNode、CommunityNode 原型 | 独立模型并增加媒体、地点与授权记录 |
| Data | 页面状态中的内存数组 | ArkData Preferences / RDB / 跨设备同步 |
| Relation | 示例关系网、新碎片动态节点与点击后生成的规则化分支 | 端侧 embedding、向量数据库与可解释关系评分 |
| Reminder | 根据通勤、睡前、周末偏好在应用内筛选一张低频回响 | Notification Kit、Form Kit 与用户授权的后台调度 |
| Community | 当前使用模拟公开碎片 | 登录、隐私授权、审核与真实社区图谱服务 |

## 3. 三屏状态机

1. **Box** 负责接收碎片、记录提醒偏好和共享范围；
2. **个人碎片轨道** 负责搜索、回顾和切换中心碎片，中心状态与关系节点都使用稳定碎片 ID；关系图中的个人节点可返回具体碎片，新写或收存的碎片会连接到原中心节点；
3. **社区延申树** 只允许从已选择匿名共享的中心碎片出发，节点按相似、相反和行动三类逐层展开。

“收进 Box”会复制社区节点为一条私密的新碎片；“继续延申”会带着当前节点的 Tag 打开输入纸条。这样不会把社区浏览做成无限信息流。

当前版本使用标签重合和规则模板，目的是先稳定验证交互。AI 后续只负责关联建议与问题生成，不擅自替用户下结论。

## 4. 隐私原则

- 每条碎片都必须明确选择匿名共享或仅自己可见；
- 私人碎片不进入社区检索与模型训练；
- 延申建议应说明连接原因；
- 社区不设计公开点赞数、粉丝排行与无限信息流。

## 5. 当前技术边界

- 数据仍保存在进程内存中，重启应用后会恢复示例数据；
- 桌面卡片、车机语音和系统通知尚未接入真实系统能力；
- 社区节点与 AI 关联为本地规则化示例，不代表真实服务返回；
- 图片、录音和视频入口等待媒体权限与持久化方案一起接入。
